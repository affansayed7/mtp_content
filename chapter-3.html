Chapter 3 — Authentication in 5G
5G Core fundamentals · quick reference
The problem with IMSI alone
Roll number analogy
Anyone can know your roll number — but knowing it doesn't make you the student.

Same with IMSI — it's not secret, anyone can copy it.
Solution — challenge-response
Network never asks: "Give me your Ki"

Instead: "Prove you know Ki"

Ki never leaves the SIM. Ever.
IMSI = Username  ·  Ki = Password  ·  Ki NEVER transmitted openly
Challenge-response flow
UE
↓
sends IMSI
Network
↓
generates random challenge
UE
calculates: f(Challenge + Ki)
↓
sends Response
Network
calculates same f(Challenge + Ki)
↓
Match → ✅ Auth OK
Mismatch → ❌ Fail
Teacher analogy
Teacher asks: "What is 47 × 89?"
Student answers correctly.
Teacher concludes student knows multiplication — without asking for the entire table.

Network asks: "Compute f(challenge)"
Only someone with the real Ki can answer correctly.
Network never asks for Ki directly — only proof that you know it
5G core functions involved
AMF
Access & Mobility Management Function
🛂 Reception desk
Receives registration request. Coordinates authentication. Does NOT store subscribers.
AUSF
Authentication Server Function
🔐 Security officer
Runs the authentication procedure. Generates challenge, verifies response.
UDM
Unified Data Management
📂 Records office
Stores IMSI, Ki, subscription info. Provides data to AUSF during auth.
Full authentication sequence
UE → Registration Request
↓
AMF — coordinates auth
↓
AUSF — requests subscriber data
↓
UDM — returns Ki + subscription info
↓
AUSF — generates challenge → UE
↓
UE — computes response using Ki
↓
AUSF — verifies response
↓
Auth success → AMF → Registration continues
Important distinction — logical vs physical storage
UDM
Logical owner of subscriber data
→
MongoDB
Physical storage in Open5GS
UDM owns the data logically — MongoDB stores it physically. Critical for debugging Open5GS.
What if Ki values don't match?
Scenario
UE Ki ≠ Network Ki

Challenge sent → UE computes wrong response → Mismatch
Result
Auth fails ✗
Registration fails ✗
PDU session never starts ✗
Everything stops at auth itself.
Auth is a gate — if it fails, nothing below (registration, PDU, internet) can happen.
Debugger mindset — when auth fails
🛠️ Initial Registration Failed → check in this order:
1. Wrong IMSI?  Check UE config
2. Wrong Ki / OPC?  Check SIM vs DB
3. Subscriber missing in DB?  Check MongoDB
4. AUSF down?  systemctl status open5gs-ausfd
5. UDM down?  systemctl status open5gs-udmd
Viva questions
Q: What is Ki?
Secret authentication key stored in SIM and network. Never transmitted.
Q: Why isn't IMSI enough for authentication?
IMSI is not secret — anyone can copy it. Ki-based challenge-response proves genuine identity.
Q: What does AUSF do?
Runs the authentication procedure — generates challenge, verifies UE response.
Q: What does UDM do?
Logically owns subscriber data (IMSI, Ki, subscription info). Physically stored in MongoDB.
Q: What does AMF do?
Handles registration and coordinates authentication — does not store subscriber data itself.
⭐ 5G uses challenge-response authentication — AUSF verifies the UE knows Ki (via computed response), UDM provides subscriber data from MongoDB, and AMF coordinates the process. Ki never leaves the SIM.
